Table Rotaion & Table Cleaner

Table rotation 试图解决的问题

  • ServiceNow 平台的某些部分（以日志记录为例）往往会创建巨大的数据集
  • 所有日志消息都写入单个表（syslog）
  • 可以合理地假设用户可能实际上想要查询此数据来解决问题

然而，这会产生一个潜在的问题 - 关系数据库往往无法在巨大的数据集上表现良好：

  • 对包含数百万（甚至数十亿）行的表执行的 SQL 查询可能会很慢
  • 一个好的数据库索引策略可以帮助解决这个问题，但只能在一定程度上 - 即使有非常好的索引，数据集最终也会变得如此之大，以至于查询性能仍然变得令人沮丧地缓慢

一个简单的解决方案就是删除旧数据以将表保持在可管理的大小，但是，这里也存在陷阱：

  • 某些数据（以电子邮件为例）根本无法删除 - 许多客户都有法律要求，这意味着这些数据必须存储多年
  • 关系数据库根本不擅长删除数据行 - 取决于如何完成它可能导致：
    1. 数据库表被锁定一段时间（某些表会立即中断）
    2. 将行标记为已删除（而不是从底层存储中物理删除）。随着时间的推移，这会导致表碎片影响查询性能，并且数据库大小缓慢但不可避免地增加

Table rotation 是怎么帮助解决这些问题的

原理与实现方式

表轮换通过 sys_table_rotation 表中的记录进行配置（可通过'System Definition -> Table Rotations'访问）。在此表中，平台中已配置表轮换的每个表都会有一行。一个重要字段是“Type” - 可以将其设置为“rotation”或“extension” - 现在让我们考虑“rotation”的含义，稍后再回到“extension”。

让我们扩展一下这意味着什么:

  • 该平台有一个名为 syslog 的逻辑表，您可以通过用户界面与该表进行交互
  • 然而，该逻辑表已配置为“rotation”类型，具有 8 次“rotations”且持续时间为 7 天
  • 这意味着底层数据库中实际上有 9 个表（基表加上 8 个“轮换”），而不是单个 syslog 表 - 请注意，在 ServiceNow 中，我们经常将这些轮换称为“shards”或“subtables”

轮换中使用的所有物理表的详细信息都保存在“sys_table_rotation_schedule”表中, 要注意的是，基表（syslog）通常不用作轮换的一部分，因此通常不会期望保存任何数据。 每个分片都有一个名为“offline”的字段，该字段设置为 true 或 false - 这指示是否可以通过用户界面查询该表中的数据。

总结 - 当配置 Table rotaion 的时候：

  • 将创建许多分片（取决于 sys_table_rotation 记录中定义的“旋转”）
  • 每个分片都会被赋予 valid_from 和 valid_to 日期/时间
  • 平台根据当前日期/时间以及每个分片的 valid_from/valid_to 将数据插入到特定分片中
  • 因此，数据根据平台写入每一行的日期/时间在逻辑上分布在分片上

然后在当轮换用完分片时

  • 其所有数据均被删除，使其为空
  • 其 valid_from/valid_to 日期将设置为未来的下周

通过执行此表旋转，始终会在当前写入位置之前保持一个分片，这意味着切换到下一个分片（当该日期/时间出现时）是无缝的。

好处

查询性能：表轮换使用分片意味着数据将在逻辑上分布在多个物理表中 - 这可以带来显着的好处

数据删除：正如之前所提到的，删除单行数据存在各种陷阱。然而，表轮换只会一次删除整个分片（或表）的数据

  • 它不需要逐行删除，只需截断相应的分片即可

Table rotation 和 extension 的区别

从根本上说，扩展与旋转极其相似：

  • 它们的定义位置与表旋转（sys_table_rotation 表）完全相同 - 但是，在本例中，旋转类型设置为“扩展”
  • 它们基于一个基表和许多分片
  • 它们被赋予一个持续时间，该持续时间决定了每个分片被写入的时间段

但请注意，表扩展中的分片不会轮换 - 扩展不会截断并重新使用最旧的分片，而只会无限期地创建新分片，从而使数据由于创建日期而在分片之间保持逻辑分离，并且不会有任何数据被删除。曾经删除过。审计 (sys_audit) 表是许多客户实例中的扩展，这意味着随着时间的推移，sys_audit 表将由大量物理分片支持。

rotation 和 extension 的抉择

  1. 如果写入的数据是暂时的/仅需要保留相对较短的时间，请使用轮换
  2. 扩展只能用于绝不能删除的数据

Table cleaner

  1. 最好适用于需要删除较少数据的表，（比如不适合 sys_audit 表）
  2. 对 table rotation 和 table extension 的表 不适用
  3. 表清理器使用“setWorkflow(false)”进行删除，这意味着可能会在记录删除时触发的业务规则、工作流和流程不会出现在表清理器的上下文中。如果您的业务逻辑依赖于此类功能，这一点很重要
  4. 它可能会花费大量时间来确定/删除记录 - 默认情况下表清理最多只花费 20 分钟从单个表中删除，因此如果查询很慢，20 分钟内删除的记录量可能会很小

表清理器是一项计划作业，默认情况下每小时运行一次 - 其目的是从实例中的表中删除较旧的记录，从而保持表大小（和查询性能）易于管理
